# TikCle-BP 데이터 구조 설명 (핸드오프용)

> 2026-06-11 코드 확인 기준. 다른 세션에 "지금 구조가 어떻게 되어있나" 설명할 때 이 문서를 기준으로.

---

## 0. 한 줄 요약

**입력(여러 업로드 경로) → 저장(관계형 테이블 = canonical / key_stats JSONB = 분석 캐시) → 화면(일부는 테이블 라이브쿼리, 일부는 key_stats 캐시).**
핵심 함정: **제품 매출만 기간(period)별로 분리 저장되고, 브랜드 KPI·크리에이터·영상·라이브는 덮어쓰기/누적이라 기간 구분이 안 된다.**

---

## 1. 데이터가 들어오는 경로 (입력)

| 경로 | 파서/액션 | 무엇을 |
|---|---|---|
| Exolyt/Apify 스크랩 | (파이프라인) | TikTok 영상 (engagement: views/likes/caption/날짜) |
| **Kalodata 화면 복붙** | `parseKalodata` | 브랜드 KPI + 제품/크리에이터/영상/라이브 표 |
| **Kalodata 크리에이터 xlsx** | `parseKalodataCreatorXlsx` | 크리에이터 상세 (Live/Video GMV 분리, 연락처) |
| **Kalodata 영상 xlsx** | `parseKalodataVideoXlsx` | 영상별 매출·광고·URL |
| **TT Shop affiliate CSV** | `parseTiktokShopUsAffiliate` | 제품별 affiliate 크리에이터 + 그들의 영상 URL |
| **BSR (Keepa)** | `uploadBsr` | Amazon BSR 시계열 |
| Helium10 / TT Shop US | (업로드) | TT Shop US 매출·affiliate raw |
| Phase 1.5 자동 스크랩 | (파이프라인) | TikTok Shop 제품 |

---

## 2. 어디에 저장되나 — 두 층

### A. 관계형 테이블 (canonical, 진짜 원본)
- **`contents`** — 모든 영상 (key = `url`). TikTok organic(엑솔릿) + 카로 영상xlsx + affiliate 영상 + IG/YT URL 다 여기. **url로 병합/dedup.**
- **`products`** — 제품 (channel = amazon / tiktok_shop). 
- **`case_product_sales`** — 제품 매출. key = `(case_id, product_id, period_end)` → **기간별 시계열.**
- **`influencers`** — 크리에이터 (key = `platform,external_id` = 핸들). `lifetime_gmv_usd` 등.
- **`sales_snapshot`** — BSR/가격 시계열 (key = `product_id, channel, collected_at`).
- **`content_clusters` + `content_cluster_members`** — 콘텐츠 포맷 클러스터 (Section C).
- **`case_video_analyses`** — 영상별 vision_tags / ASR / pass3_meta_id.
- **`ig_posts` / `yt_videos` / `ig_authors`** — IG/YT 데이터.

### B. `cases.key_stats` (JSONB) — 분석 결과 **캐시 스냅샷**
- `phase2` (월별 추이·영상수·creator 수·sales_summary), `phase3`(티어), `phase4b_clusters`, `phase4c`/`phase4d`, `phase5`(BSR inflection·USP).
- `kalodata_brand`, `kalodata_videos`, `kalodata_lives`, `kalodata_videos_xlsx`, `kalodata_creators_xlsx`.
- `tt_shop_us_affiliates`, `tt_shop_us_helium10`, `phase1_5`.
- ⚠️ **key_stats는 분석 돌린 시점의 스냅샷이라 contents가 커지면 stale(낡음)이 된다.**

---

## 3. 병합/우선순위 규칙 (가장 헷갈리는 부분)

| 데이터 | key | 같은 게 또 들어오면 |
|---|---|---|
| 영상 (contents) | `url` | 병합 (url 같으면 덮어씀) |
| 제품 매출 (case_product_sales) | `case_id+product_id+period_end` | **기간 다르면 별도 행 (시계열 보존)** |
| 크리에이터 (influencers) | 핸들 | **덮어쓰기 (마지막 업로드 이김)**. 단 영상xlsx 경로만 insert-only |
| 브랜드 KPI (kalodata_brand) | — | **non-null 덮어쓰기 (마지막 기간 이김)** |
| 영상/라이브 (kalodata_videos/lives) | caption / title+start | **누적 (dedup만)** |
| BSR (sales_snapshot) | product+collected_at | 제품 단위로 **전체 삭제 후 재삽입** |

### 복붙 vs xlsx 분업 (안 싸움)
- **복붙(화면 텍스트)** = 브랜드 레벨 요약. 영상은 `key_stats.kalodata_videos`로만 가고 **contents엔 안 들어감 → 분석에 안 잡힘.**
- **영상 xlsx** = `contents`에 insert → **영상수·추이·클러스터·Section D per-video GMV에 실제로 쓰이는 알맹이.**
- **크리에이터 xlsx** = 상세(연락처·GMV분리) 별도 저장.

## 4. 분석 파이프라인 (phase, 결과는 key_stats에 캐시)

`phase1_5`(TT Shop 스크랩) → `phase2`(SQL 집계: 추이/영상수/creator/sales) → `phase3`(티어) → `phase3.5/3.7` → `phase4a`(광고) → `phase4b.1`(샘플) → `4b.2`(ASR) → `4b.3`(vision 태깅) → **`4b.4`(3-pass 클러스터링)** → `4b.5`(SKU 매칭) → `phase4c`(IG) → `phase4d`(YT) → `phase5`(포지셔닝·BSR inflection).

- 재실행 시 **캐시된 phase는 skip** (force_phases로 강제 재실행).
- ⚠️ **4b.4 클러스터링이 OOM/타임아웃 단골** (pass1을 순차→병렬화로 근본 해결함). cleanup이 기존 클러스터를 먼저 지우므로, 중간에 터지면 클러스터 손상 → 재실행 전 백업 권장.
- ⚠️ **TikTok 클러스터링은 vision_tags 필요** (IG/YT는 caption 폴백). vision 없이 돌면 TK가 클러스터에서 빠짐.

## 5. 화면(케이스 페이지)이 뭘 읽나

| 섹션 | 소스 | 비고 |
|---|---|---|
| **A 콘텐츠 활동** | 추이=**contents 라이브 집계로 패치**(phase2 stale 보정), BSR=sales_snapshot | 영상수/추이는 라이브 |
| **B 인플루언서 풀** | **contents 라이브쿼리**(allTkCreators) + influencers GMV + 티어(phase3/서버SQL) | 토글/요약/표 = 같은 풀 길이로 통일 |
| **C 콘텐츠 포맷** | **content_clusters 테이블** (key_stats 비어도 복원) | 클러스터 = 테이블이 진짜 |
| **D 매출 & BSR** | products + case_product_sales + sales_snapshot + kalodata_brand/videos_xlsx | SKU 매칭 = 영상 제목 ↔ SKU명 |

원칙: **일부는 테이블에서 라이브로 읽고(A 추이·B 풀·C 클러스터), 일부는 key_stats 캐시.** 라이브로 읽는 건 데이터 추가하면 바로 반영, 캐시는 재실행해야 갱신.

---

## 6. 알아둘 함정 (이번 세션에서 확인된 것)

1. **period(기간)는 각 복붙의 "MM/DD/YYYY ~ MM/DD/YYYY" 한 줄에서만 읽음.** 이전 복붙 기억 안 함. 없으면 **오늘 날짜로 박힘.** → 섹션만 따로 붙이면 날짜 줄 꼭 포함.
2. **제품 매출만 기간별 분리.** 브랜드 KPI·크리에이터는 덮어쓰기, 영상·라이브는 누적 → "기간별 브랜드 매출" 같은 건 현재 구조로 안 됨.
3. **phase2는 스냅샷** → 스크랩 더 쌓이면 영상수/creator가 낡은 수치. (Section A는 라이브로 패치해둠, 다른 데는 재실행 필요.)
4. **복붙 영상은 contents에 안 들어가서 분석 안 됨.** 영상 분석하려면 **영상 xlsx 필수.**
5. **PostgREST 기본 1000행 제한** — 큰 쿼리는 range 페이지네이션 안 하면 일부 누락 (BSR·creator 등에서 버그 났었음).
6. **카로 영상은 엑솔릿이랑 같은 TikTok 영상을 양쪽에서 수집** → 단순 합산하면 더블카운트. contents에서 url(=video id)로 dedup한 게 진짜 고유 수.
