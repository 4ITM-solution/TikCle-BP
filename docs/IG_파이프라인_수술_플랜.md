작성: 2026-06-10 · 대상: TikCle-BP `run-analysis.ts` 파이프라인
목적: 오늘 겪은 "A·B·C에 IG 안 뜸 + 하나 고치면 다음 버그" 체인을 **근본에서** 끝낸다.

## 한 줄 요약

> **버그는 사실상 2개다.** ① IG/YT 스크랩이 그걸 쓰는 phase보다 **늦게** 돈다. ② 클러스터를 **삭제 없이 쌓는다.**

## 근본 원인 (확정)

### 뿌리 ①  IG/YT 스크랩이 소비 phase보다 늦음
- 위 표가 전부. phase2(A)·phase4b(C)가 4c(IG 스크랩) **앞**에서 돌아 IG를 0으로 본다.
- **이게 오늘 고생의 ~80%.** A 안 뜸, C 안 뜸이 전부 이 한 원인.

### 뿌리 ②  phase4b-clusters가 옛 클러스터를 안 지움
- `src/lib/inngest/aggregators/phase4b-clusters.ts` line 118/143/189: `content_clusters` / `content_cluster_members`에 **insert만, delete 없음.**
- 재실행할 때마다 클러스터가 **쌓임** → 6769b0bb는 **content_clusters 91개**(여러 런 누적), 멤버는 15개 클러스터에만, 화면 메타클러스터는 또 다른 셋 → **메타↔멤버 미스정렬** → C 채널 토글이 0.
- 오늘 per-case force가 케이스를 **꼬이게 만든 진짜 메커니즘**이 이것.

### 뿌리 ③  phase2 캐시가 IG/YT 변화에 무효화 안 됨 (①의 보조)
- `run-analysis.ts` line 281: `if (existing.phase2 && !force("phase2") && !phase1_5_New)` → IG 새로 들어와도 **재집계 안 함.**
- 그래서 "IG 스크랩 후 한 번 더 돌리면 되겠지"도 안 통하고 → 매번 `force phase2` 수동.

### 별개 ④  ad-heavy 케이스 phase4a 타임아웃
- `phase4a.ts` `hybrid` 모드 = curious 스크랩 + partnership **200건 detail 재호출**. 685광고 케이스에서 **800s(maxDuration) 초과** → `http_unreachable` → 분석 전체 실패 → IG까지 못 감.
- 오늘 6769b0bb 첫 실패의 원인(내 재호스트 회귀가 악화시켰지만, 베이스 자체가 무거움).

### 별개 ⑤  Meta 썸네일 재호스트 (현재 없음)
- E섹션은 `meta_ads` 테이블 썸네일(FB CDN)을 보여줘서 시간 지나면 엑박.
- 내가 인라인 683개 재호스트 시도 → 타임아웃 → **되돌림(revert)**. 지금은 재호스트 없음(원상태).

## 수술 항목 (우선순위 · 각각 독립 · 하나씩 테스트→배포)

### 🥇 Fix ②  클러스터 delete-before-insert  *(리스크 낮음 / 가치 큼)*
- **무엇**: `runPhase4bClusters` 시작에서 이 케이스의 기존 `content_clusters`(+ `content_cluster_members`) 삭제 후 insert.
- **왜 먼저**: ①과 독립이고, 누적 91개 정리 = 미스정렬·꼬임 해소. 재실행이 멱등(idempotent)해짐.
- **리스크**: 낮음 (delete→insert 표준). FK/cascade 순서만 확인(멤버 먼저 or ON DELETE CASCADE).
- **테스트**: 1개 케이스 phase4b force 1회 → `content_clusters` count가 **누적 안 되고 일정**한지 확인.

### 🥈 Fix ①+③  IG/YT 스크랩을 phase2·phase4b **앞**으로  *(리스크 중간 / 핵심)*
- **무엇** (권장 = "스크랩만 당기기", 최소 변경):
  1. `runPhase4c`(스크랩+ig_posts insert) **호출만** phase1.5 직후로 이동. 결과는 변수에 보관.
  2. `phase-4c-save`(stats 저장)는 **현 위치(맨 끝) 유지** → cascade-save 충돌 회피 그대로.
  3. phase4d도 동일.
  4. 캐시 무효화 추가: `phase2New`·`phase4bSampleNew` 조건에 **"4c/4d가 새로 스크랩됨"** 포함.
- **효과**: IG 넣고 **분석 한 번**이면 phase2(A)·phase4b(C)가 IG 보고 채움. **force 불필요 → 꼬임 없음.**
- **리스크**: 중간 (스크랩 step 위치 이동 + 캐시 조건). 저장 블록은 안 건드리니 cascade는 안전.
- **테스트**: IG config 있는 1개 케이스 깨끗이 1회 실행 → A IG 월별 + C IG 클러스터 **둘 다** 첫 런에 뜨는지.
- **대안(더 안전하지만 비쌈)**: 4c/4d 후 phase2+phase4b 2nd-pass 재실행 → vision 비용 2배라 **비권장.**

### 🥉 Fix ④  ad-heavy 자동 curious  *(리스크 낮음)*
- **무엇**: phase4a에서 fetch된 광고 수 > 임계(예: 400)면 `hybrid`의 200 detail 재호출 **스킵**(curious-only).
- **효과**: 685광고 케이스도 phase4a가 800s 안에 완료.
- **리스크**: 낮음 (조건 분기 1개). partnership 정보만 일부 손실(대형 케이스 한정).
- **테스트**: 6769b0bb(685광고) 분석 → phase4a `http_unreachable` 안 나고 완료되는지.

### 4️⃣ Fix ⑤  Meta 썸네일 **배치** 재호스트  *(리스크 낮음 / 우선순위 낮음 = 미용)*
- **무엇**: 인라인 ❌. `phase-4b-vision` 처럼 **작은 청크(step.run N개)로 나눠** meta_ads 썸네일을 storage 재호스트.
- **리스크**: 낮음(이미 검증된 배치 패턴). 단 비용/시간만 신경.
- **우선순위 낮음**: 엑박은 기능 아니라 미용. ①②④ 후에.

## 안전 프로세스 (오늘의 교훈 반영 — 이걸 꼭 지킨다)

1. **한 번에 하나.** Fix ② → 확인 → Fix ① → 확인 → … 묶어서 배포 ❌.
2. **테스트 케이스 1개에서 먼저 검증.** "배포하고 사용자가 버그 발견" ❌ → "내가 먼저 1케이스로 확인하고 보고" ✅.
3. **1 fix = 1 배포 = 1 확인.** 연속 배포(청크 캐시 깨짐) 금지.
4. **force 겹쳐 쏘지 않기.** 한 케이스에 런 진행 중이면 끝나고. (오늘 꼬임의 원인)
5. **문제 생기면 즉시 revert.** (오늘 phase4a 회귀처럼 — revert는 항상 가능.)
6. **재실행 ≠ 화면 잠김.** status=running이어도 기존 데이터는 보임 → 사용자 작업 안 막힘.

# Part 2 · 섹션별 "질문 → 답" 매핑 (UX 리스트럭처 기준)

> 문제의식(사용자): "한 섹션에 내용이 늘어지는데, 이걸로 **내가 뭘 이해해야 하는지** 모르겠다."
> 해결 원칙: **모든 섹션·탭은 정확히 하나의 질문에 답해야 한다.** 답이 안 나오는 블록(예: 매칭 영상 0인 BSR 변곡점)은 노이즈 → 빼거나 합친다.

## 이 도구가 답하는 최상위 질문
> **"이 브랜드는 (인플루언서 시딩으로) 어떻게 성공했는가? 우리가 따라 하려면 뭘 베껴야 하는가?"**
이걸 5개 하위 질문으로 쪼갠 게 A~E다. 각 섹션은 그 하위 질문 1개의 답이어야 한다.

| 섹션 | 답하는 하위 질문 (한 문장) |
|---|---|
| **A 콘텐츠 활동** | *언제* 터졌나? — 시딩 모멘텀의 **타임라인** (시점·채널·티어 구성의 시간 변화) |
| **B 인플루언서 풀** | *누구*를 썼나? — 동원한 **인플 풀의 구조** (티어 믹스·재계약·cross-channel·shop creator) |
| **C 콘텐츠 포맷** | *무엇*을 찍었나? — 먹힌 **콘텐츠 앵글·포맷·USP** |
| **D 매출 & BSR** | *그래서 팔렸나?* — 시딩이 만든 **매출·순위 성과** (SKU·영상·크리에이터별) |
| **E Meta 광고** | *유료는 어떻게?* — **paid 채널·랜딩·partnership** 전략 |

## A · 콘텐츠 활동  → "언제, 어떤 채널에서, 누가 터뜨렸나?"

| 블록(현재) | 답해야 할 질문 | 보여줄 답 | 결정 (✅ 사용자 확정 2026-06-10) |
|---|---|---|---|
| 채널 토글 (전체/TK/IG/YT) | 어느 채널이 주력이었나? | 채널별 영상 수 + 추이 | 유지. 토글 (0) 비활성 버그는 Part 1에서 해결 |

**리스트럭처 방향**: A의 통합 차트는 **유지(겹침이 핵심 가치)**. 손볼 것 = ① 변곡점 카드를 **매칭→날짜윈도우 전체영상**으로 ② 게시빈도 분포 B로 이관 ③ "hero SKU 기준" 라벨.

**🐛 BUG / UX (✅ 사용자 확정 2026-06-10)** — *"전반적으로 뭘 눌러야 하는지 모르겠다"*:

- **제품 이름이 너무 길어 앞부분만 보임 → 무슨 제품인지 모름** → 직관적 제품 식별 방법 필요. 후보: 짧은 별칭/핵심어 추출 · 썸네일 · 호버 시 풀네임 · "PDRN Gua Sha"처럼 키워드만.
