# 케이스 상세 프로토타입 지시서 — 실패 4회의 교훈 포함

## 왜 이 지시서가 있나 (직전 세션의 실패 기록 — 반복 금지)
1. 정적 스케치 → 반려 ("호버·토글 없으면 판단 불가")
2. 간소화 프로토 → 반려 ("배포판보다 정보가 적은 걸 어떻게 검수하나")
3. 차트 주인공을 추정으로 교체 → 반려 ("티어×월이 주인공인데 오가닉/광고를 그림")
4. 스펙 문서·그렙 기반 재구성 → 반려 ("**배포판이 왜 그렇게 구성됐는지 이해를 안 함**")

공통 원인: **컴포넌트 코드를 실제로 정독하지 않고 문서·기억·상상으로 재구성.**

## 절대 순서 (건너뛰기 금지)
1. **정독**: `src/app/cases/[id]/page.tsx`(3,649줄) + `src/components/case-detail/mockup/` 6개(6,093줄)를 **전부 읽는다**. 각 블록마다: 순서 / 형태(카드·표·차트·임베드) / 데이터 필드 / 토글·클릭 동작 / 조건부 렌더 조건을 인벤토리 표로.
2. **이해 보고**: "배포판은 왜 이 순서·형태인가"를 블록별 한 줄로 — 사용자가 이 구성에 이미 익숙하고 그 배치 자체가 사용자 선택의 기록임을 전제로.
3. 그 위에 개선안: 변경하는 블록만 [제안] 칩, 근거 한 줄. **배치·형태의 기본값은 배포판.**
4. 클릭되는 HTML (PROTOTYPE_PROTOCOL.md 전 규율 + node --check).
5. 사용자 루프: 반려는 그 자리에서 수정, 같은 주소 재게시.

## 이미 확정된 것 (재론 금지)
- A 타임라인 주인공 = **티어×월 스택** (광고집행·BSR은 보조 라인, %토글 유지)
- 광고 섹션 = **소재 이미지 카드가 1급** (Storage 재호스트 썸네일), 표는 보조
- 픽스처 = 전 축 데이터가 채워진 것 (필요시 합성 + 정직 라벨)
- 결론 한 줄·드릴다운↗·용어 사전·표본/신선도 라벨 문법 (v5까지 무반려 통과분)
- 리디자인 아님 — 배포판 + 개선. 간소화 절대 금지.
