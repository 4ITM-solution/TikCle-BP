# 화면 설계 프로토콜 — "설명 문서"가 아니라 "클릭되는 HTML 프로토타입"

> 모든 화면 신설·개편은 이 프로토콜을 따른다. FE 레인의 구현 착수 조건 = **프로토타입 확정**.

## 1. 진단 먼저 — 상상 금지
- 대상 화면의 **실제 컴포넌트 코드**를 읽고, 데이터는 **실제 쿼리**로 확인해서 진단한다.
- 문제는 원인 층(데이터/권한/화면)을 구분해서 보고한다. (QA 레인의 3층 추적과 동일 규율)

## 2. 화면 정의서 형식 (화면마다)
1. 이 화면이 답하는 질문 **한 줄**
2. 목업
3. 구성 요소 표: 요소 / 보여주는 것 / 누르면 어디로
4. 핵심 카드 해부도: 목업에 ①②③ 번호 → 옆 표에서 번호별 정의
- 화면 간 이동은 **유저 플로우 맵** (화면 박스 → 트리거 버튼 칩 → 도착 화면)
- **용어 사전 필수** — 화면에 영문·내부 코드·전문용어 노출 금지 (U3와 동일)

## 3. 확정 전에 반드시 클릭되는 프로토타입
- self-contained HTML 하나 (외부 리소스 금지, vanilla JS)로 **실제 동작**까지:
  버튼 누르면 항목이 빠지고, 드로어가 열리고, 토스트가 뜨게. 데이터는 **실측값**.
- 사용자가 만져보고 피드백 → 그 자리에서 고쳐 **같은 주소로 재게시** — 이 루프를 짧게 여러 번.
- **코드 구현은 프로토타입 확정 후에만.**

## 4. 설계 규율 (전 화면 공통)
- **한 사건은 화면에 한 번만** — 사실 1곳 + 행동 1곳, 중복 섹션 통합
- **모든 집계 숫자에 출처·기준시각·구성 내역 드릴다운(↗)** 부착
- **빈 화면은 다음 행동을 말한다** — 왜 비었고 뭘 하면 채워지는지
- **판단은 사람, 화면은 근거** — 자동 처리는 유일 매칭만, 애매하면 후보를 보여주고 사람이 고른다
- **무시/삭제는 사라지지 않는다** — 보관함 + 복원 (기록 보존)
- **목록 항목 구조 통일**: [유형 칩] + 내용 + [연결 대상 칩(클릭→상세)] + 근거 줄 + 액션

## 5. 확정 후
- 프로토타입을 `docs/design/prototype/`에 **정본으로 커밋** → **1:1 구현 (재해석·축소 금지)** → 구현되면 실화면 실측으로 검증.

## 규율 추가 (2026-07-11 반려에서 증류)
- **프로토타입은 축소본 금지** — 기존 화면이 있으면 그 블록 인벤토리(코드 전수 기반)에서 출발, 전 블록 포함. 바꾸자는 것만 [변경 제안] 칩으로 병기. "대표 블록만 그린 문법 시연"은 검수가 성립하지 않음 (사용자가 "없어진 것 찾기"부터 해야 함).
- **재게시 전 스크립트 문법 검사 필수** (`node --check`) — TIERS 중복 선언으로 차트 전체 사망 실사고.
