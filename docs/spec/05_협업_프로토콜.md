# 05 협업 프로토콜 — 세션/개발자 공통 규칙

## 1. 역할 구조

```
오케스트레이터 (메인 세션, 사용자와 대화하는 창)
  설계·지시서 발행·검수·머지·Supabase 마이그레이션 적용·Vercel 배포. 구현하지 않음.
실행 세션 (별도 터미널 창, 보통 Sonnet)
  지시서 1개 = 세션 1개. 코드 작성과 로컬 커밋까지만.
```

## 2. 실행 세션 규칙 (전원 필수)

1. 시작하면 `docs/spec/README.md` → 제품정의(BP_재설계_v2 §1) → 자기 지시서(`docs/ws/WSn_지시서.md`) 순으로 읽는다.
2. 브랜치 `ws-n-<이름>`에서 작업. **git push · Vercel 배포 · Supabase 마이그레이션 적용 · 케이스 데이터 삭제 절대 금지.**
3. 스키마 변경은 `supabase/migrations/0NN_*.sql` 파일 작성까지만 (다음 번호 사용, 적용은 오케스트레이터).
4. 유료 API(LLM·Apify) 실호출은 지시서에 명시된 소량 검증만.
5. `npx tsc --noEmit` 통과 후 완료 처리.
6. 완료 시: `docs/ws/WSn_REPORT.md` 작성 (변경 파일 · 결정/해석 사항 · **배포 전 사람이 확인할 것** 목록) + `docs/BP_재설계_v2.md` §6에 한 줄 로그.
7. 지시서에 없는 판단이 필요하면: 임의로 확장하지 말고 REPORT에 "결정 필요" 항목으로 기록.

## 3. 오케스트레이터 게이트 (머지→배포 순서)

1. REPORT 검수 → 브랜치 머지 (충돌 시 설계 문서 기준으로 해소)
2. 마이그레이션 있으면 Supabase 적용 (dry-run 성격의 검증 SQL 먼저)
3. `git push origin main` = Vercel 배포. **1 WS = 1 배포 = 1 검증** (연속 배포 금지)
4. 검증: 지시서의 완료 기준 실측 (SQL/화면/케이스 1개 실행) → §6 로그
5. 문제 시 즉시 revert — revert는 항상 가능

## 4. 파이프라인 수동 조작 (오케스트레이터/운영용)

```bash
# 개별 phase 재실행 (force=캐시 무시)
KEY=$(grep 'INNGEST_EVENT_KEY=' .env.local | head -1 | cut -d'"' -f2)
curl -s -X POST "https://inn.gs/e/$KEY" -H 'Content-Type: application/json' \
  -d '{"name":"case/phase.requested","data":{"case_id":"<uuid>","phase":"<phase>","force":true}}'
# phase: collect-ttshop|collect-meta|collect-ig|collect-yt|enrich-creators|enrich-ig-profiles|
#        interpret-asr|interpret-tag|interpret-cluster|interpret-sku|serve-stats
# 전체 분석: {"name":"case/start.analysis","data":{"case_id":"<uuid>"}}
# 상태 확인: select * from phase_runs where case_id='<uuid>';
```
