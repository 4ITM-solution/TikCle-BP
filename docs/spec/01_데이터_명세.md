# 01 데이터 명세 — 테이블·뷰·JSONB 사전 + 적재 경로

> 2026-07-07 프로덕션 DB(dxjodlxkynjirldpumxr) 실측 기준. 신규 세션은 이 문서만으로 "어떤 데이터가 어디에 어떻게 쌓이는지"를 파악할 수 있어야 한다.
> 상위 문서: `docs/BP_재설계_v2.md` (제품 정의·아키텍처). 감사 결과: `docs/ws/DATA_감사_2026-07-07.md`.

## 1. 스코프 규칙 (제일 먼저 이해할 것)

| 스코프 | 테이블 | 규칙 |
|---|---|---|
| **brand+country** | contents, products(구형 일부), sales_snapshot | 케이스와 직접 FK 없음. `cases.brand_id + cases.country`로 조인해서 케이스에 귀속 |
| **case_id** | ig_posts, ig_authors, yt_videos, yt_channels, meta_ads, content_clusters, case_video_analyses, case_product_sales, phase_runs | 케이스 직속 |
| **글로벌** | influencers, brands | 케이스 간 공유. ⚠️ influencers는 여러 케이스가 같은 행을 갱신 — 필드 소유권 규칙(§4) 준수 |

권역 케이스(MENA 등) 주의: contents.country는 케이스 country와 일치해야 조인됨. 제품 country는 세부국가일 수 있음 (알려진 함정, BP_재설계_v2 P9).

### ⚠️ 죽은 테이블 (전부 0행 — WS5에서 drop 또는 구현 결정)

`sales_monthly, internal_notes, campaign_executions, pipeline_runs, promotion_events, viral_bsr_impacts, viral_clusters, case_rejections, case_video_assets, app_settings, seeding_packages`

- **pipeline_runs 0행**: "이력 추적" 테이블인데 실제로 한 번도 안 쓰임. phase_runs가 실질 대체 — WS5에서 drop 권장.
- **promotion_events 0행**: §1.0.2 완결성 ⑥축(시점)이 필요로 하는 테이블인데 비어있음 → **WS4에서 프라임데이 등 메가 프로모션 캘린더 시드 필요**.

### 🔒 보안 특이사항 (Supabase advisor 경고)

- RLS 꺼짐 4개: `case_product_sales, case_video_analyses, case_video_assets, app_settings`.
- 단, 나머지 테이블도 전부 "anon full access" 정책이라 **실질 노출 수준은 동일** (내부 도구 + anon key 전제 설계). 외부 공개 전환 시 전면 재설계 필요 — 현 시점 조치: RLS enable + anon-all 정책으로 통일해 advisor 경고만 정리 (WS5).

## 7. 알려진 함정 (기존 문서에서 상속 — 신규 세션 필독)

1. Kalodata **복붙** 영상은 contents에 안 들어감 → 분석 대상은 **영상 xlsx** 업로드분.
2. period는 복붙 텍스트의 "MM/DD/YYYY ~" 줄에서만 파싱 — 없으면 오늘 날짜로 박힘.
3. PostgREST 기본 1000행 제한 — 대량 조회는 range 페이지네이션 필수.
4. Kalodata 영상과 Exolyt 영상은 같은 TT 영상 양쪽 수집 → contents url dedup된 수가 진짜.
5. tiktok_shop_scraper의 매출(정가×누적)은 최대 144배 과장 — 매출은 Kalodata/Helium만 신뢰 (메모리 검증).
6. BSR은 sales_snapshot이 유일 소스 (case_product_sales.bsr·key_stats는 stale 가능).
